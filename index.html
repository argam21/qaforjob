Types of Testing

Functional Testing

Checks that software does what it should. Each feature works as requirements describe. Example: clicking "Buy" places the order correctly.

Non-Functional Testing

Checks HOW WELL the software works. Performance, security, usability, reliability. Example: the Buy button works, but does it respond under 1 second when 1000 users click simultaneously?

Manual vs Automated

Manual: A human tests by clicking, typing, exploring. Best for new features, exploratory testing, usability, and complex business scenarios.

Automated: Scripts run tests automatically. Best for repetitive regression. Faster but cannot find UX issues or anything not scripted.

Regression Testing

Re-test existing features after changes to confirm nothing broke. Done after every build or release. Can be full or partial (impacted areas only).

Smoke Testing

Quick basic checks to confirm a build is stable enough for deeper testing. Like checking if a car starts before testing the brakes. Usually 30-60 minutes. If smoke fails, the build goes back to development.

Sanity Testing

Narrower than smoke. Quick check of a specific fix or small feature. Confirms the fix works without full regression.

Integration Testing

Tests that two or more parts of the system work together. Example: after sign-up, does the welcome email service receive the trigger?

End-to-End (E2E) Testing

Tests a complete user flow from start to finish across all systems. Example: open account, deposit, place trade, close trade, withdraw.

Exploratory Testing

Tester explores the app without a predefined script. Uses experience and intuition. Senior QAs find the most unexpected bugs this way. Not random — it is structured exploration using charters.

Black Box / White Box / Grey Box

Black box: No access to internal code. Tests from the user's perspective.

White box: Uses internal code knowledge. Common in unit and integration testing.

Grey box: Mix of both. Senior QAs typically work in grey box mode.

UAT (User Acceptance Testing)

Testing done by business stakeholders (not QA) to confirm the product meets business requirements before go-live. QA supports by setting up environments and test data. UAT sign-off is usually a release gate.

Shift-Left Testing

Involving QA earlier in the process. QA reviews specifications and designs before development starts — not just tests at the end.

Why it works: Bugs found in requirements cost almost nothing to fix. Bugs found in production cost 100x more.

Real example: A QA Proactive Board where QA reviews specs before dev writes a single line of code. Questions like "what happens if the user has no balance?" are answered in requirements, not discovered as production bugs. Result: near-zero major production incidents over 18 months.

QA Role in Releases

Before

  • Verify all P1 test cases passed
  • Confirm all P1/P2 bugs fixed or have approved workarounds
  • Run final regression suite
  • Sign off release notes, confirm rollback plan

During

  • Monitor dashboards (error rates, response times, crash rates)
  • Run smoke tests in production immediately after deployment

After

  • Watch user-reported issues and support tickets
  • Support hotfix or rollback if needed
  • Postmortem if anything broke. Update regression suite with gaps found.
Senior QA mindset: "I bring data, the business makes the call." Document everything in writing. Never block a release on opinion — block on data.

Test Design Techniques

Structured methods to decide WHAT to test. Senior QAs apply these automatically.

Equivalence Partitioning (EP)

Divide inputs into groups where all values behave the same way. Test one value from each partition.

Example: Age field, valid 18-65.

PartitionRangeTest valueExpected
Below valid0-1715Error
Valid18-6535Accepted
Above valid66+70Error

Boundary Value Analysis (BVA)

Most bugs live at the edges. Test the minimum, maximum, and values just inside and outside each boundary.

ValueTypeExpected
17Just below minimumError
18Minimum (valid)Accepted
19Just above minimumAccepted
64Just below maximumAccepted
65Maximum (valid)Accepted
66Just above maximumError
Use EP to find partitions, then BVA to test the edges of each. They work together.

Decision Table Testing

When behavior depends on combinations of conditions, map every combination to its outcome.

ConditionC1C2C3C4
Account funded?YesYesNoNo
Market open?YesNoYesNo
Allow order?YesNoNoNo

Each column is one test case. Full combination coverage in 4 tests.

State Transition Testing

Map all states and transitions. Each transition is a test case. Also test invalid transitions (can a Filled order be cancelled? It should not be).

Example — Order lifecycle: Draft → Submitted → Processing → Filled. Also: Submitted → Cancelled, Processing → Failed.

Pairwise Testing

Ensures every pair of variables is tested at least once. Most bugs are triggered by 1-2 variables interacting. Pairwise gives high coverage with far fewer test cases.

OS (Win/Mac) x Browser (Chrome/Firefox/Safari) x Connection (WiFi/4G) = 12 full combinations. Pairwise = 6 tests covering all pairs.

Error Guessing

Experience-based. Common targets: empty inputs, zero, negative numbers, very long strings, special characters ' " < > & /, concurrent actions, interruptions (network drop mid-request), first/last items in lists.

Defect Metrics

  • Production incident rate: Bugs escaping to production per release. The most important metric.
  • Defect leakage: Production bugs that should have been caught in testing.
  • Reopen rate: High rate signals dev quality or QA retest process issue.
  • Bug aging: Long-open bugs are a risk signal.
Never track "bugs found per QA engineer." It incentivizes trivial bug reporting. The right KPI is production incident rate.

Entry and Exit Criteria

Formal checkpoints defining when testing can START and when it is DONE. They protect QA from being pressured into starting too early or shipping too quickly, and turn release decisions into data-driven checklists instead of gut feelings.

Entry criteria = conditions that must be true BEFORE testing begins.
Exit criteria = conditions that must be true BEFORE the product ships.

Entry Criteria

Minimum conditions that must be met before QA begins. Testing before these are met wastes time — you find environment problems, not product bugs.

Typical entry criteria checklist

  • Build deployed to test environment successfully
  • Development smoke tests passed (basic build stability confirmed)
  • All features in scope are implemented and code-reviewed
  • Test environment configured correctly (correct data, correct configs)
  • Test data prepared and available (user accounts, seeded data, mocked services)
  • Required test devices and browsers available
  • Acceptance criteria documented and signed off
  • Previous blocking defects from last cycle resolved

Real example: Trading Platform Feature Release

Feature: New leverage settings screen

Entry Criteria:
1. Build v2.6.0 deployed to staging
2. Dev smoke test passed: screen loads without crash
3. AC-201 through AC-215 (leverage specs) finalized
4. Test accounts with all account types available in staging
5. Mock trading server responding
6. Design spec v1.3 attached to the Jira ticket

QA will NOT begin testing until all 6 are confirmed.
Starting testing without entry criteria means spending half a day debugging an environment issue that dev should have caught — and your test report reflects that unfairly.

Exit Criteria

What must be true before testing is complete and the release can proceed.

Typical exit criteria checklist

  • All P1 test cases executed and passed
  • All P2 test cases executed — pass or approved-skip with documented risk
  • Zero open P1 defects
  • Zero open P2 defects, OR each has an approved workaround documented in writing
  • P3/P4 defects reviewed — fixed, deferred with PM approval, or accepted as known issues
  • Regression suite passed, meeting the agreed threshold (e.g. 98%+)
  • Performance baselines verified (response times within SLA)
  • UAT sign-off from business stakeholders (if applicable)
  • Release notes written and reviewed
  • Rollback plan confirmed with DevOps
  • QA sign-off formally recorded (Jira, Confluence, or email)

Real example: Mobile App Release

Release: iOS v4.2 — Login and Account Features

Exit Criteria:
1. 100% of P1 test cases executed and passed
2. 95%+ of P2 test cases passed
3. No open bugs with severity S1 or S2
4. All S3 bugs reviewed:
   - 2 deferred to v4.3 (PM approved in writing)
   - 1 accepted as known issue (logged in release notes)
5. Full regression suite: 143/145 pass
   (2 skipped — known flaky, tracked in backlog)
6. Devices tested: iPhone 14 (iOS 17), iPhone 12 (iOS 16),
   iPhone SE (iOS 15)
7. Login response time < 2 sec confirmed on 4G
8. UAT completed by Product team on June 20
9. Release notes approved by PM
10. Rollback to v4.1 confirmed with DevOps (< 5 min)

QA sign-off recorded in Confluence Release Tracker.

When Exit Criteria Are Not Met

QA presents the data. The business decides. Three outcomes:

  • Block the release: Postpone until fixed. Right choice for critical failures.
  • Ship with known issues: PM formally accepts the risk, issues are documented, support team is briefed. QA records this in writing.
  • Partial release: Ship what is ready, disable the broken feature via feature flag.
Senior answer: "QA's job is to present data clearly — what passed, what failed, what the risk is. The go/no-go decision belongs to the business. But I make sure it's a documented, informed decision. If the PM ships over my objection, that conversation is in writing."

Suspension and Resumption Criteria

Sometimes testing must pause mid-cycle due to a blocker. Suspension criteria say when to stop. Resumption criteria say when to restart.

Suspension Criteria (pause testing if):
- A P1 bug blocks access to the features being tested
- Test environment becomes unstable or unavailable
- More than 30% of P1 test cases failing
  (build is likely not test-ready)

Resumption Criteria (restart when):
- Blocking P1 bug is fixed and verified in staging
- Environment restored and confirmed stable
- Dev confirms a new build with fixes is deployed

Test Data Management

Types of test data needed

  • Happy path: Valid users, funded accounts, correct configurations
  • Edge case: Zero balance, maximum values, expired accounts, locked accounts
  • Negative: Invalid formats, missing required fields, duplicate entries
  • Boundary: Values at the exact edges of valid ranges
  • Permission: Users with different roles — admin, read-only, banned

Test data challenges

  • Data decay: Test accounts expire, seeded data gets modified by other testers
  • Data conflicts: Multiple QAs sharing the same test accounts cause interference
  • Privacy: Never use real production user data in test environments without anonymization
  • Data dependency: Test B depends on data created by Test A — if A fails, B fails too

Best practices

  • Each QA owns dedicated test accounts
  • Critical tests set up their own data and clean up after
  • Production data is anonymized before copying to staging
  • Data reset scripts exist to restore environment to a known state

The Mental Model

Table users:

idusernameagecountrybalance
1argam36Armenia1500
2john25USA200
3maria42Spain5000
4ali30Dubai800
5anna28Armenia100

TABLE = the box. COLUMN = a category. ROW = one record. Primary key = unique identifier per row (the id column).

SELECT and FROM

SELECT * FROM users                    -- all columns, all rows
SELECT username, country FROM users    -- specific columns only
SELECT without FROM is an error. Always need both.

WHERE

SELECT * FROM users WHERE country = 'Armenia'
SELECT * FROM users WHERE balance > 500
SELECT * FROM users WHERE country = 'Armenia' AND balance > 500
SELECT * FROM users WHERE country = 'Armenia' OR country = 'Spain'
Single quotes for text ('Armenia'). No quotes for numbers (500).

ORDER BY and DISTINCT

SELECT * FROM users ORDER BY balance DESC         -- highest first
SELECT * FROM users ORDER BY age ASC              -- youngest first
SELECT * FROM users ORDER BY country ASC, balance DESC
SELECT DISTINCT country FROM users                -- no duplicates
SELECT COUNT(DISTINCT country) FROM users         -- count unique

GROUP BY and Aggregates

SELECT country, COUNT(*) AS user_count FROM users GROUP BY country
SELECT country, SUM(balance) AS total FROM users GROUP BY country
SELECT country, AVG(balance) AS avg FROM users GROUP BY country
SELECT MAX(balance) AS highest, MIN(balance) AS lowest FROM users
With GROUP BY, SELECT can only contain: (1) the grouped column, (2) aggregate functions. COUNT(*) cannot go in WHERE. SUM(*) does not exist — use SUM(column_name).

HAVING

SELECT country, COUNT(*) AS user_count
FROM users GROUP BY country
HAVING COUNT(*) > 1    -- only countries with more than 1 user

WHERE filters rows BEFORE grouping. HAVING filters groups AFTER GROUP BY.

JOIN

Table orders: id, user_id, amount, status.

JOIN produces a temporary combined result. It does not create a permanent table. The table in FROM is the LEFT table.
-- INNER JOIN: only rows with matches in both tables
SELECT users.username, orders.amount
FROM users
INNER JOIN orders ON users.id = orders.user_id
-- Returns argam and maria (they have orders). Others excluded.

-- LEFT JOIN: all left table rows, NULL if no match
SELECT users.username, orders.amount
FROM users
LEFT JOIN orders ON users.id = orders.user_id
-- Returns ALL users. john/ali/anna get NULL for amount.

-- Classic QA: find users who never placed an order
SELECT users.username FROM users
LEFT JOIN orders ON users.id = orders.user_id
WHERE orders.id IS NULL

NULL Handling

NULL is the absence of a value. You CANNOT use = to check for it.
WHERE country IS NULL      -- Correct
WHERE country = NULL       -- Wrong, always returns 0 rows
WHERE country IS NOT NULL  -- Correct

LIKE, IN, BETWEEN, LIMIT

-- LIKE: % = any characters, _ = exactly one character
WHERE username LIKE 'a%'         -- starts with a
WHERE username LIKE '%ar%'       -- contains ar

-- IN: shortcut for multiple OR
WHERE country IN ('Armenia', 'Spain', 'USA')

-- BETWEEN: inclusive
WHERE age BETWEEN 25 AND 35      -- same as >= 25 AND <= 35

-- LIMIT with OFFSET for pagination
SELECT * FROM users ORDER BY username LIMIT 10 OFFSET 20  -- rows 21-30

Query Execution Order

  1. FROM
  2. JOIN
  3. WHERE
  4. GROUP BY
  5. HAVING
  6. SELECT
  7. ORDER BY
  8. LIMIT

Why you cannot use a SELECT alias in WHERE (WHERE runs before SELECT), but can in ORDER BY.

What is an API

Application Programming Interface. How two software systems talk to each other.

Restaurant analogy: You (client) order food. The kitchen (server) makes it. The waiter (API) carries the order and brings the result. You never see inside the kitchen.

Authentication Types

  • API Key: Key in header or query param. Simple. If leaked, anyone can use it.
  • Basic Auth: Username:password base64-encoded. Old. Only acceptable over HTTPS.
  • Bearer Token (JWT): Authorization header. Most common. Contains encoded user info + expiry.
  • OAuth 2.0: "Sign in with Google." App gets a token, never sees the user's password.

What to Test in Any API

Functional

  • All methods work as documented
  • Response body matches documented schema
  • Correct status codes for each scenario

Validation and edge cases

  • Missing required fields → 400 with clear error
  • Wrong data types → 400
  • Boundary values: max length, zero, negative numbers
  • Empty arrays, null values

Auth and authorization

  • No token → 401
  • Expired/invalid token → 401
  • Valid token, wrong role → 403
  • User A accessing User B's resource → 403 (IDOR vulnerability if it returns 200)

Error handling

  • Errors have clear human-readable messages
  • Errors do not expose stack traces or internal paths (security risk)
  • Consistent error format across the whole API

Performance and security

  • Response time within SLA
  • Rate limiting works (429 after N requests)
  • All endpoints use HTTPS
  • Sensitive data never in URLs
  • SQL injection and XSS payloads handled safely

Idempotency

MethodIdempotent?Why

Test: Send the same POST twice. Check whether one or two records are created. A properly implemented API should handle retries without creating duplicates.

Concurrency Testing

User has $100. Two browser tabs place a $100 order simultaneously. Expected: one succeeds, one fails with "Insufficient balance." Race condition bug: both succeed, balance goes to -$100.

Test using Postman Collection Runner with parallel iterations, or two simultaneous curl/script calls.

Rate Limiting and Pagination

Rate limiting tests

  • Normal speed → all succeed
  • N+1 requests in the window → 429
  • Wait for window reset → succeed again
  • Response includes Retry-After header

Pagination tests

  • First page returns correct count (e.g. 10 items)
  • Last page returns remaining items, not empty results
  • Total count in response is correct
  • Requesting beyond last page → empty list, not an error

Postman: Tests, Variables, Chaining

// Status code
pm.test("Status is 200", () => pm.response.to.have.status(200));

// Response time
pm.test("Time < 500ms", () => pm.expect(pm.response.responseTime).to.be.below(500));

// Field exists
pm.test("Has order_id", () => {
    pm.expect(pm.response.json()).to.have.property('order_id');
});

// No sensitive data leaking
pm.test("No password in response", () => {
    pm.expect(pm.response.json()).to.not.have.property('password');
});

// Save token from login for next request
const token = pm.response.json().access_token;
pm.environment.set("token", token);
// Next request header: Authorization: Bearer {{token}}

Swagger / OpenAPI

Standard format to document REST APIs. Most modern APIs have a Swagger UI to see all endpoints and try them live.

When given a Swagger URL and asked to test it:

  1. Read endpoint description and purpose
  2. Identify required vs optional parameters
  3. Try with valid data → verify 200 and response matches schema

  6. Remove auth token → expect 401
  7. Try another user's resource ID → expect 403
  8. Check documented error codes match what the API actually returns

Senior-Level Insights

Questions that separate seniors from juniors. Know these cold.

1. REST is a convention, not a law

Common violations you will encounter

  • POST returns 200 instead of 201
  • GET /orders/456/cancel actually cancels the order (GET must never modify data)
  • POST /createOrder instead of POST /orders (action URL vs resource URL)
  • Returns 200 with {"success": false, "error": "..."} instead of proper error code
Senior answer: "REST is a convention, not an enforced protocol. APIs vary widely. As QA I check two things: what documentation says, and what the API actually does. The gap between them is exactly where bugs live."

2. GET is less secure than POST — and POST is not automatically secure

Why GET leaks

  • URL logged in server logs, proxy logs, CDN logs, analytics tools
  • URL ends up in browser history, can be bookmarked or shared
  • Referrer header sends the URL to third parties
  • CSRF: an image tag with a malicious GET URL in its src triggers the action automatically

Why POST is not automatically secure

  • POST body is still readable in network traffic without HTTPS
  • POST does not provide encryption — HTTPS does that
  • POST does not prevent CSRF — CSRF tokens do that
Senior answer: "GET puts data in the URL which gets logged across infrastructure. POST puts it in the body. But real security needs HTTPS for encryption, authentication tokens for access control, CSRF tokens for form protection, and input validation."

3. Does an API work offline? (the trick question)

  • Web/HTTP API: Needs internet. Cannot complete offline.
  • OS API: Camera, GPS, file system — work fully offline.
  • Library API: Local functions. No internet needed.

Modern offline-first apps queue HTTP requests locally and sync when connection returns. From the user's perspective it seems to work offline.

Senior answer: "It depends on which kind of API. The strongest move is to ask 'Do you mean web/HTTP APIs or APIs in general?' — clarifying before answering is itself a senior-level response."

4. Race conditions and concurrent updates

  • Without locking: Last write wins. One update silently overwrites the other.
  • Optimistic locking: Record has a version number. If it changed since you read it, update rejected with 409. User refreshes and retries.
  • Pessimistic locking: Record locked while one user edits. Others wait.

Critical in trading and banking. Test by sending two conflicting requests simultaneously.

5. Other curveball questions

Does a website work without JavaScript?

SPAs (React, Angular, Vue): completely broken. Server-rendered sites: mostly works. Progressive enhancement: basic functionality works, JS adds features.

Is HTTPS enough for security?

No. HTTPS encrypts transit only. Does not prevent SQL injection, broken auth, weak passwords, XSS, CSRF, IDOR, business logic flaws.

Authentication vs authorization

Authentication = WHO are you? Authorization = WHAT can you do? You can be authenticated but not authorized.

Can API testing replace UI testing?

No. API covers backend logic. UI catches visual bugs, browser rendering, UX, accessibility. They complement each other.

If a test passes 100 times, can you fully trust it?

Not necessarily. Could have environmental dependencies, be testing the wrong thing, or consistently masking a bug. Trust comes from test design quality, not repetition.

When do you stop testing?

You make a risk-based decision. Testing stops when exit criteria are met, risk is acceptable, and the cost of more testing exceeds the expected benefit. QA provides data; the business makes the go/no-go call.

Common QA Interview Mistakes

What NOT to say. These patterns immediately signal a junior mindset, even from experienced candidates.

Mistake 1: "My job is to find bugs"

Your job is to reduce risk and provide information that enables good decisions. Finding bugs is a side effect, not the goal. Optimizing for bug count leads to useless test cases.

Better: "My job is to provide accurate information about product quality so the team can make informed release decisions."

Mistake 2: "I test everything"

Exhaustive testing is impossible. Every interviewer knows this. Saying it signals either a lack of understanding or dishonesty. Risk-based prioritization is the correct answer.

Better: "I prioritize test coverage based on business risk, change impact, and historical defect patterns."

Mistake 3: Measuring quality by bug count

More bugs can mean a bad product or good testing. Fewer bugs can mean high quality or shallow testing. The right metrics: production incident rate, defect leakage, coverage of critical paths.

Mistake 4: Confusing severity and priority

Asked constantly, failed constantly. Severity = technical impact (QA sets it). Priority = business urgency (PM sets it). They are completely independent. A cosmetic bug can be high priority. A crash can be low priority. Know the difference cold.

Mistake 5: Vague bug reports

"Login doesn't work" or "Something is wrong on the home page" signal a junior immediately. Every report needs: clear title, steps to reproduce, expected vs actual, environment, evidence. A developer should reproduce it from the report alone without a single follow-up question.

Mistake 6: Not asking clarifying questions

When given a "test this" task in an interview — a screen, feature, or API — the first thing a senior QA does is ask: What is the purpose? Who are the users? What are the acceptance criteria? What is the most critical flow? Jumping straight into test cases without context is a junior pattern.

Mistake 7: "Automation will replace manual testing"

Automation handles repetitive regression well. It cannot handle exploratory testing, usability, new features, or unscripted scenarios. Senior QAs know both have permanent roles and that they complement each other.

Mistake 8: Not knowing your own resume

If you listed a tool, framework, or methodology, be ready to explain it in detail with a real example. Listing "Selenium" without explaining how a test is structured, or listing "Agile" without knowing what happens in a sprint, are immediate red flags.

Testing Pyramid

  • Unit tests (bottom — most tests): Written by developers. Test individual functions in isolation. Fast, cheap, many.
  • Integration tests (middle): Test that components work together. More complex, moderate speed.

Scrum Ceremonies — QA Role in Each

Sprint Planning

The team selects user stories from the backlog for the sprint. QA's role:

  • Review acceptance criteria of each story — flag ambiguities before development starts
  • Estimate testing effort for each story
  • Flag dependencies: "This story requires a test account type that doesn't exist yet"
  • Identify stories that need performance or security attention
Sprint planning is where shift-left starts. Questions caught here cost nothing. The same question found as a bug in production costs 100x more.

Daily Standup (Daily Scrum)

15-minute daily sync. Three questions: what did I do yesterday, what am I doing today, any blockers?

QA-specific standup patterns:

  • "Yesterday: tested story X, found 2 bugs (logged). Today: completing retest of story Y after dev fix."
  • If blocked: "Blocked — the build for story Z isn't deployed to staging yet." (This is data, not complaint.)
  • Senior QAs also mention risk: "I'm concerned we won't finish testing story W before the sprint ends — it's more complex than estimated."

Sprint Review (Demo)

Team demonstrates completed work to stakeholders. QA's role:

  • Confirm what is actually "done" meets the definition of done (DoD)
  • Sometimes demo the testing approach or show the test coverage for a feature
  • Provide a quality summary: "3 stories completed and fully tested. 1 story deferred — 1 P2 bug remaining."

Sprint Retrospective

Team reflects on the process: what went well, what didn't, what to improve. QA contributions:

  • Flag patterns: "We had 3 regression bugs this sprint because feature X touched the payment module — we need better integration coverage there."
  • Propose process improvements: "If QA gets stories 2 days before sprint end instead of 1 day, we can do deeper testing."
  • Acknowledge wins: "The new entry criteria checklist prevented 2 wasted days of testing a broken build."

Backlog Refinement (Grooming)

Team reviews upcoming stories before they enter a sprint. QA's most valuable ceremony.

  • Review ACs for completeness and testability
  • Ask edge case questions: "What happens if the network drops mid-payment?"
  • Identify missing ACs and get them added to the story before development starts
  • Flag stories that are too vague to test: "This AC says 'the form should work correctly' — what does correctly mean?"

Definition of Done (DoD)

A shared agreement on what "done" actually means for a story to be considered complete. Without a DoD, "done" means different things to different people.

Example DoD for a user story

  • Code reviewed and approved by at least one other developer
  • Unit tests written and passing
  • Feature deployed to staging environment
  • QA has executed all test cases for the story
  • No open P1 or P2 bugs
  • Acceptance criteria verified and signed off by QA
  • Feature works on supported browsers / devices
  • Release notes updated if user-facing change
If a story doesn't meet the DoD, it is NOT done — no matter what the developer says. QA is the guardian of the DoD.

QA in Agile: Common Senior Interview Questions

"How do you handle testing when development isn't finished until the last day of the sprint?"

"This is a process problem, not a testing problem. The fix is definition of done that requires stories to be in staging X days before sprint end. I raise this in retrospectives with data: 'In the last 3 sprints, stories entered staging with less than 1 day left. This caused 2 carry-overs and 1 production bug.' Showing the pattern gets it fixed."

"How do you keep up with fast sprints — new features every 2 weeks?"

"Three things: (1) risk-based prioritization — I don't test everything equally. (2) shift-left — I catch issues in refinement before they're written as bugs. (3) regression automation for stable areas — frees manual testing time for new features."

"What's the difference between a story and a bug in your backlog?"

A user story describes new functionality to be built. A bug describes a defect in existing functionality. Both get tracked, estimated, and prioritized — but they go through different workflows. Stories have ACs; bugs have reproduction steps and expected vs actual results.

Defect Trend Analysis

Tracking bug trends over time reveals process health. Senior QAs present trends, not just point-in-time counts.

  • Rising bug count per release: Product complexity growing faster than test coverage, or code quality declining
  • High reopen rate: Dev is not reproducing bugs before marking them fixed, or QA retest criteria are unclear
  • Bugs concentrated in one module: That module needs refactoring or dedicated regression attention
  • Production bugs increasing: Regression suite is missing critical scenarios — review and expand it

Accessibility Testing

Accessibility testing (a11y) is increasingly expected from senior QAs. It verifies that the product can be used by people with disabilities — visual, hearing, motor, or cognitive. It is also legally required in many markets (WCAG compliance).

Why Accessibility Matters

  • Legal requirement in many countries (ADA in US, EN 301 549 in EU, Equality Act in UK)
  • ~15% of the global population has some form of disability
  • Accessibility fixes often improve usability for everyone (captions help in noisy environments, keyboard navigation helps power users)
  • Inaccessible products are increasingly rejected in enterprise procurement

What to Test — Practical Checklist

Keyboard navigation

  • Tab through the entire page — every interactive element should be reachable
  • Tab order should be logical (left-to-right, top-to-bottom)
  • Focus indicator must be visible at all times (the highlight around the focused element)
  • All actions achievable with keyboard: open menus, submit forms, close modals
  • No keyboard trap: pressing Tab should never get stuck inside a component

Screen reader testing

  • All images have meaningful alt text (or empty alt="" if purely decorative)
  • Buttons have descriptive labels: "Submit order" not just "Submit"
  • Form fields have associated labels (not just placeholder text)
  • Error messages are announced by screen reader when they appear
  • Dynamic content changes (modals opening, toast messages) are announced

Tools: VoiceOver (Mac/iOS built-in), TalkBack (Android built-in), NVDA (free Windows screen reader), JAWS (enterprise Windows).

Color and contrast

  • Text contrast ratio minimum: 4.5:1 for normal text, 3:1 for large text (WCAG AA)
  • Information is never conveyed by color alone (e.g. "required fields in red" must also have an asterisk or label)
  • UI works in high contrast mode (Windows/iOS high contrast settings)

Tools: Colour Contrast Analyser, browser DevTools (Accessibility tab shows contrast ratios).

Text and zoom

  • Page is usable at 200% browser zoom — no content hidden or overlapping
  • Text can be resized without breaking layout
  • No horizontal scrolling required at standard viewport sizes

Forms

  • Every input has a visible, programmatically associated label
  • Required fields are clearly marked (not just color)
  • Error messages identify which field has the error and how to fix it
  • Autocomplete attributes set correctly (e.g. autocomplete="email")

Images and media

  • Informative images: alt text describes the content and purpose
  • Decorative images: empty alt attribute (alt="")
  • Videos: captions available for all spoken content
  • Audio: transcript available
  • No content that flashes more than 3 times per second (seizure risk)

Performance and Load Testing

Performance testing verifies the system behaves correctly under expected and unexpected load. It is one of the most common senior QA interview topics — and the most common follow-up question is: "Where do those numbers come from?"

Where Do Performance Numbers Come From?

This is the question that separates seniors from juniors. Every performance test needs target numbers — but interviewers want to know how you define them, not just that they exist.

Source 1: Business-defined SLAs (best case)

The product team or business stakeholders define Service Level Agreements. These are formal commitments about performance that the system must meet.

Examples of business-defined SLAs:
- "Login must complete in under 2 seconds for 95% of requests"
- "Order placement must respond in under 500ms at peak load"
- "The platform must support 10,000 concurrent users without degradation"
- "API uptime must be 99.9% (less than 8.7 hours downtime per year)"

When SLAs exist, your job is to test against them and report pass or fail. This is the cleanest scenario.

Source 2: Production baseline (most common in practice)

No formal SLA? Measure what production currently does, then use that as your baseline. The rule: staging must perform at least as well as production, ideally better.

How to get baseline numbers:
1. Check APM tools (Datadog, New Relic, Grafana)
   - Average response time for key endpoints
   - 95th percentile response time (p95)
   - Peak concurrent users from the last 30-90 days
   - Error rate at peak load

2. Example baseline you might find:
   - Production login p95: 1.2 seconds
   - Peak concurrent users: 820 (from analytics)
   - Error rate at peak: 0.3%

3. Your test targets become:
   - Staging login p95 must be <= 1.2 seconds
   - Test with 1000 concurrent users (peak + ~20% headroom)
   - Error rate must stay under 1% under load
Always add 20-25% headroom on top of peak production numbers. If production peaks at 800 users, test for 1000. You want to know the system can handle growth before growth happens.

Source 3: Industry UX standards (fallback when no data exists)

When there are no business SLAs and no production data (new product, new feature), use research-backed industry benchmarks as your starting point. These are defensible in any interview or stakeholder conversation.

Real Performance Test Example (k6)

// k6 load test — login endpoint
import http from 'k6/http';
import { check, sleep } from 'k6';

export let options = {
  stages: [
    { duration: '2m', target: 100 },   // ramp up to 100 users
    { duration: '5m', target: 100 },   // hold at 100 users
    { duration: '2m', target: 500 },   // ramp to 500 (stress)
    { duration: '2m', target: 0 },     // ramp down
  ],
  thresholds: {
    http_req_duration: ['p95<2000'],   // 95% of requests under 2 seconds
    http_req_failed: ['rate<0.01'],    // error rate under 1%
  },
};

export default function () {
  const res = http.post('https://staging.example.com/api/login', {
    email: 'testuser@example.com',
    password: 'TestPass123',
  });
  check(res, {
    'status is 200': (r) => r.status === 200,
    'response time OK': (r) => r.timings.duration < 2000,
  });
  sleep(1);
}

This test ramps to 100 users, holds, then spikes to 500. It will fail if p95 exceeds 2 seconds or error rate exceeds 1%. These thresholds come from the agreed SLA.

Performance Testing in QA's Role vs Dev's Role

In many companies, performance testing sits in a grey zone. Senior QAs should be clear about this:

  • QA owns: defining performance test scenarios, setting thresholds based on SLAs, running tests, interpreting results, reporting pass/fail, recommending action
  • Dev owns: investigating and fixing performance bottlenecks identified by QA
  • DevOps/Infra owns: infrastructure scaling decisions based on test findings
  • Product/Business owns: defining the SLAs that QA tests against
Senior answer when asked about your role in performance testing: "I own the test design, execution, and reporting. I define scenarios based on production traffic patterns and business SLAs, run the tests, and present findings with clear pass/fail against thresholds. I don't fix the bottlenecks — that's engineering. But I make sure the data is clear enough that the right people can act on it quickly."

Client-Server Architecture

Almost every modern application is built on the client-server model. Understanding it helps QA think about where bugs live, what to test at each layer, and how to diagnose failures.

The model

A client is anything that requests data or a service. A server is anything that provides it. The client initiates the conversation. The server responds.

Client examplesServer examples
Web browser (Chrome, Safari)Web server (Nginx, Apache)
Mobile app (iOS, Android)API server (Node.js, Django, Spring)

How a typical web request works

  1. User types qaforjob.netlify.app in the browser
  2. Browser asks DNS: "What is the IP address for this domain?" (more on DNS below)
  3. DNS responds with the IP address
  4. Browser sends an HTTP request to that IP address
  5. Server receives the request, processes it, sends back a response
  6. Browser renders the HTML/CSS/JS and displays the page

Why this matters for QA

  • Where is the bug? A failure could be in the client (UI bug), the API (logic bug), the database (data bug), or an external service (third-party bug). Understanding the layers helps you narrow it down fast.
  • What to test at each layer: Client = UI, UX, rendering. API = logic, validation, auth. Database = data integrity, constraints. External = integration, error handling when service is down.
  • Simulating failures: What happens when the database is slow? When the payment gateway is down? When the cache is stale? Senior QAs test these scenarios.
Senior interview answer when asked "where would you look if a feature is broken?": "I start by checking the client-side network tab to see what request was made and what response came back. If the request looks correct but the response is wrong, the bug is on the server. If the request itself is malformed, the bug is in the client code. If the response is correct but the UI shows something wrong, it is a rendering bug. The network tab is always my first stop."

Stateless vs stateful

REST APIs are stateless -- each request contains all the information the server needs. The server does not remember previous requests. Session information is sent with every request (via token or cookie). This is why JWT tokens exist -- they carry user identity with every request so the server does not need to store session state.

HTTP vs HTTPS

HTTP (HyperText Transfer Protocol) is the protocol browsers and servers use to communicate. HTTPS is HTTP with encryption layered on top via TLS (Transport Layer Security).

Real examples of why HTTP is dangerous

Example 1 -- Login on public WiFi

User connects to coffee shop WiFi and logs into a site over HTTP. Anyone on the same WiFi network running a packet sniffer (like Wireshark) can see the username and password in plain text. With HTTPS, they see encrypted garbage that is useless without the decryption key.

Example 2 -- Man-in-the-middle attack

A malicious router intercepts HTTP traffic between the user and the bank. It modifies the page on the fly -- changing the account number in a transfer form. The user sees the original page, but the data goes to the attacker. HTTPS prevents this because any modification breaks the cryptographic signature and the browser rejects it.

Example 3 -- ISP snooping

An Internet Service Provider can see every HTTP request a user makes -- which pages they visit, what they search for, what data they submit. With HTTPS, the ISP can only see that the user connected to a domain -- not what they did there.

Example 4 -- Session hijacking

Over HTTP, session cookies are transmitted in plain text. An attacker on the same network can steal the cookie and impersonate the logged-in user -- even if the login page itself was secure. HTTPS encrypts the cookie in transit, preventing this.

TLS Handshake (simplified)

When you connect to an HTTPS site, a handshake happens before any data is sent:

  1. Browser says: "Hello, here are the encryption methods I support"
  2. Server responds: "Hello, here is my SSL certificate and my chosen encryption method"
  3. Browser verifies the certificate is valid and trusted
  4. Both sides agree on an encryption key
  5. All further communication is encrypted with that key

This whole process takes milliseconds and happens automatically.

SSL certificate

An SSL/TLS certificate proves the server is who it claims to be. It is issued by a trusted Certificate Authority (CA) like Let's Encrypt, DigiCert, or Comodo. When you see the padlock in the browser, it means: (1) the connection is encrypted, and (2) the certificate has been verified.

Common QA test: What happens when the SSL certificate expires? The browser shows a scary warning: "Your connection is not private." Users cannot proceed without clicking through a warning. This is a P1 production issue -- test certificate expiry handling in your environments and set up alerts before it happens.

What QA should test for HTTPS

  • All pages load over HTTPS -- no mixed content warnings (HTTP resources on an HTTPS page)
  • HTTP URLs automatically redirect to HTTPS (301 redirect)
  • SSL certificate is valid, not expired, and matches the domain
  • Sensitive data (passwords, tokens, card numbers) never appear in HTTP requests
  • HSTS header is set (tells browsers to always use HTTPS for this domain)
  • API endpoints all use HTTPS -- no HTTP fallback

DNS (Domain Name System)

DNS is the internet's phone book. It translates human-readable domain names (like qaforjob.netlify.app) into IP addresses (like 104.21.45.22) that computers use to find each other.

Why computers need this

Computers communicate using IP addresses -- numbers, not names. But humans remember names, not numbers. DNS sits in the middle, translating one to the other automatically every time you visit a website.

How DNS resolution works

You type: qaforjob.netlify.app

1. Browser checks its local cache
   "Have I looked this up recently?" If yes, use cached IP. Done.

2. OS checks its local cache + hosts file
   Still not found? Ask a DNS resolver.

3. DNS Resolver (usually your ISP or 8.8.8.8)
   Asks the Root DNS server: "Who handles .app domains?"

4. Root DNS server
   "Ask the .app TLD nameserver"

5. TLD nameserver (.app)
   "Ask Netlify's nameserver for netlify.app"

6. Netlify's authoritative nameserver
   "qaforjob.netlify.app = 104.21.45.22"

7. DNS resolver caches the answer and returns it to your browser

8. Browser connects to 104.21.45.22 over HTTPS

TTL (Time To Live)

Every DNS record has a TTL -- how long (in seconds) other DNS servers can cache the answer before asking again. Short TTL (300 seconds) = changes propagate fast but more DNS queries. Long TTL (86400 = 24 hours) = cached longer, fewer queries, but changes take longer to propagate globally.

Real QA scenario: You deploy a new server and update the DNS A record. But some users still see the old site for hours. Why? Their DNS resolver cached the old IP address and the TTL has not expired yet. This is called DNS propagation delay. Always lower TTL to 300 seconds BEFORE making DNS changes, then restore it after. Senior QAs know to warn the team about this during release planning.

Why DNS matters for QA

  • Environment routing: Staging and production often use DNS to route traffic. A misconfigured DNS record can send production traffic to staging or vice versa -- a serious bug.
  • SSL certificate validation: Certificates are issued per domain. If DNS is wrong, the certificate will not match and HTTPS will fail.
  • Testing after domain changes: After a DNS change, flush your local DNS cache (sudo dscacheutil -flushcache on Mac) to test the new routing without waiting for TTL expiry.
  • CDN and load balancing: Many companies use DNS-based load balancing -- different users get different IP addresses. This can cause hard-to-reproduce bugs if the servers are not in sync.
  • Subdomain testing: staging.example.com, api.example.com, cdn.example.com all rely on correct DNS records. Always verify these point to the right places after any infrastructure change.
